# PharmaScanner (医薬品 GS1 スキャナー) 開発履歴・チェンジログ

## プロジェクト概要
iPhone等のスマートフォンを用いて、医薬品に印字されたGS1バーコードをスキャンし、Google Apps Script (GAS) バックエンドと連携して「在庫数」「棚番」「最終納品日」を即座に照会するWebアプリケーション (PWA) です。

---

## 開発セッション履歴 & 技術的挑戦 (2026-03-24)

### 1. 初期構築とGASバックエンド統合
- 既存の堅牢なGAS（Looker Studio/MedOrder連動等）プロジェクトに対し、GS1コード（14桁GTIN）からYJコードへの変換ロジック `searchMedicineByGS1` を追加実装しました。
- `Code_Merged.gs` として既存コードを一切破壊せずにマージし、既存の受信POSTロジックの中に `action === 'search_gs1'` の受け口を新設しました。
- スプレッドシート上に、GTINとYJコードを紐付ける「変換マスター」タブを作成する仕様を策定しました。

### 2. PWAフロントエンド開発（カメラ仕様の変遷と限界突破への挑戦）
iPhone (Safari) 上で動作する純粋なカメラスキャナーアプリの開発において、技術的な壁と以下の変遷を経ました。

- **第1期（html5-qrcode標準搭載版）**
  - Web標準のZXingを用いたカメラスキャナーを実装。
  - **課題**: 100錠箱のGS1-128やDataMatrixはスキャンできるものの、**PTPシートや小瓶に印字される極小バーコード「GS1 DataBar Limited / Stacked」は、オープンソースのWebアプリ用解析エンジン（ZXing）では規格非対応のため物理的に読み取れない**ことが判明しました。

- **第2期（ネイティブ・BarcodeDetector API検証版）**
  - 外部ライブラリの限界を突破するため、iOS 17以降のSafariに隠されているApple純正の強力なShape Detection API (BarcodeDetector) の直接起動を試みました。
  - **課題**: デフォルトでFeature Flagsがオフになっており、またOSへの制約（Autoplay Policyやカメラの解像度制約によるOverconstrainedError）が厳しく、一部の環境で起動失敗が多発しました。

- **第3期（ZBar WebAssembly実装版）**
  - GS1 DataBarを確実に読み取るため、C++で作られたエンジンをWebAssembly化してブラウザに埋め込むという高度な技（`ZBar WASM`）を導入。
  - **課題**: モジュールロードのCORSエラーや、古いiOSでの非対応によるサイレントクラッシュ（無言のフリーズ）が発生し、すべてのデバイスでの安定稼働に支障が出ました。

### 3. 最終解決策：AsReader (ASR-020D) ハードウェア連携版の完成
- **結論**: ブラウザのカメラ機能単体でのGS1 DataBarの高速・安定読み取りの限界を認め、ユーザー環境で利用可能な高性能バーコードリーダージャケット **「AsReader ASR-020D」を用いた物理スキャナ（HID・キーボードエミュレーション）方式** に完全シフトする英断を下しました。
- **実装内容**:
  - 全てのカメラ制御コード、WASM、外部ライブラリを完全パージし大幅な軽量化を実現。
  - 画面中央に常にフォーカスが当たるシンプルな `<input>` フォームを設置。
  - AsReaderがトリガー入力で流し込む「GTIN + 改行(Enter)」をJavaScriptの `submit` イベントで瞬時に（0.1秒以内）キャッチする構成に変更。
  - AsReaderが設定によって付与する不要な接頭辞（`]C1`、`]d2` 等）や括弧を完全に削除する強力な正規表現フィルターを標準搭載。

### 4. SDKモードへの進化展望（AsBrowserの活用）
その後のセッションにおいて、AsReader を最大限に活用する手法として「SDKモード（シリアル・専用通信）」の導入および運用プランを検討・策定しました。

- **背景**: Safari等の一般Webブラウザ（現在のHIDモード）では、「文字を入力するためのテキストボックスに常にフォーカスが当たっていなければならない」「入力時にソフトウェアキーボードが無駄に飛び出す」という、UI運用上のボトルネックが存在します。
- **究極の解決策**: iOSアプリストアから提供されている無料Webブラウザ **「AsBrowser（アズブラウザ）」** を使用することで、Web画面のままSDKモードに完全対応できるようになります。
- **SDKモード (AsBrowser) の圧倒的メリット**:
  1. 画面の裏側（専用のJavaScript関数）に直接読み取りデータがコールバックされるため、そもそも「入力枠」が不要になります。
  2. ソフトウェアキーボードが起動しなくなり、画面全体を使った美しく広大なWebアプリUIが構築できます。
  3. 処理速度・ラグがさらに短縮され、真の意味でのハンディターミナル化がWebアプリで完結します。
- **将来の拡張性**: 当PWAがAsBrowser上で運用されることを見越し、現在の `app.js` の `processScannedCode(raw)` 関数をグローバルな窓口として3行開けることで、**HIDモードとSDKモードの両方にシームレスに同時対応できる最強のアーキテクチャ**として設計されています。

## 結論と今後の運用
本アプリは、カメラの起動待ちやピント合わせのラグがなく、PTPシートの DataBar Limited だろうと何だろうとハードウェアのレーザー性能で100%読み取れる**「AsReaderに最適化された超高速・最強のWebターミナル」**として完成しました！

いつでもこの `CHANGELOG.md` を参照することで、なぜWebアプリ単体でのカメラ活用を見送りハードウェア連携にシフトしたのか、という技術的背景を振り返ることができます。
